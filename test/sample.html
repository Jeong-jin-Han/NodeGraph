Main topic Problem: RM checkpointing의 한계 ▲
Recommendation model 학습에서 embedding table은 수십~수백 TB 규모.
기존 checkpointing은 학습 중단 + 전체 상태 저장이 필요하여 overhead가 심각.
장애 발생 시 복구에도 오랜 시간 소요.
Original §1, p.1
Training recommendation models (RMs) that contain massive embedding tables spanning tens of terabytes requires frequent checkpointing, which significantly degrades training throughput and increases recovery time upon failures.
📄 원문 §1
Main topic CXL-based Architecture 제안 ▲
CXL Type-2 디바이스 두 종류를 제안:
- CXL-GPU: MLP 연산 담당
- CXL-MEM: Embedding 연산 + checkpointing 담당

두 디바이스가 cache-coherent domain을 형성하여 호스트와 일관된 메모리 접근 가능.
Original §3.1, p.5
We propose TrainingCXL, which introduces two CXL Type-2 devices: CXL-GPU for MLP operations and CXL-MEM for embedding operations with integrated checkpointing capability, forming a cache-coherent domain.
Method Batch-aware Checkpoint 메커니즘 ▼
Undo logging 기반 checkpoint.
RM의 특성상 embedding indices를 미리 알 수 있어서 update 대상을 사전에 파악 가능.
이를 활용하여 background에서 checkpoint 수행.
Original §4.1, p.9
Since the sparse features include the indices of target embeddings, the checkpoint manager can identify which entries will be updated before the actual training computation begins.

Transformer, RL 등에서는 이 가정이 성립하지 않음.

The batch-aware checkpoint mechanism relies on the RM-specific property that embedding indices are known prior to computation.
Question 왜 embedding table이 수십~수백 TB인가? ▼
RM은 user ID, item ID, context 등 수십억 개의 sparse categorical feature를 각각 embedding vector로 매핑.

예: Meta DLRM
- Category 수: ~10억
- Embedding dim: 64~128
- 단일 테이블: ~500GB
- 전체 모델: 수~수십 TB
Original §2.1, p.2
Modern RMs maintain embedding tables with billions of entries, each mapping a categorical feature to a dense vector representation.

공통점: 둘 다 CAMEL Lab, persistence를 투명하게.

Question CXL Type-2가 정확히 뭔가? ▲
CXL 디바이스 3종류:
- Type-1: Accelerator (자체 캐시, 호스트 메모리 사용)
- Type-2: Accelerator + 자체 메모리 (CXL-GPU, CXL-MEM이 이것)
- Type-3: Memory expander (연산 없이 메모리만 제공)

Type-2는 자체 프로세싱 + 자체 메모리를 모두 가져서 embedding 연산을 offload할 수 있음.
